## Техническое задание: дашборд аналитики потребления Neon по проектам

### 1. Цель и контекст

**Цель:** единый дашборд, где видно **по каждому проекту Neon** расход ресурсов (метрики биллинга/usage) **по дням и месяцам**, с **ежедневным** автоматическим сбором данных и хранением в **PostgreSQL на Neon**. Фронт и API — приложение на **Vercel**.

**Пользователь:** вы (владелец организации Neon с несколькими проектами).

---

### 2. Источник данных (Neon)

- Использовать **официальный Neon API** для метрик потребления на уровне **организации / проектов** (Consumption / usage API). В документации Neon это описано как получение **истории потребления по проектам** (метрики вроде compute, storage, transfer и т.д. — в зависимости от плана и версии API).
- **Аутентификация:** Personal API Key или ключ с нужными правами из Neon Console (Bearer token).
- **Идентификаторы:** в ТЗ зафиксировать, что в конфиге хранятся минимум: `NEON_API_KEY`, **`org_id`** (идентификатор организации в Neon), при необходимости — список `project_id` или правило «все проекты орг».
- **Ограничения плана:** на Free/некоторых планах детальная история потребления по API может быть недоступна или ограничена — в реализации предусмотреть **явную ошибку/баннер** в UI и логирование, если API возвращает 403/пустые данные.

_Ссылка для разработчика:_ [Neon API — consumption / usage](https://api-docs.neon.tech/) и гайды вроде [consumption metrics](https://neon.tech/docs/guides/consumption-metrics) / [orgs API consumption](https://neon.tech/docs/manage/orgs-api-consumption).

---

### 3. Хранение данных (PostgreSQL на Neon)

Отдельная БД (или отдельная схема) **только для аналитики** — не смешивать с прод-данными приложений без необходимости.

**Минимальная модель (логическая):**

| Сущность                  | Назначение                                                                                                                                                                                                    |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `projects` (кэш)          | `neon_project_id`, имя, опционально метаданные с последней синхронизации                                                                                                                                      |
| `usage_snapshots`         | Снимок за **конкретный день** (или интервал): `date` (UTC), `neon_project_id`, набор числовых полей по метрикам (compute seconds, storage bytes, transfer bytes и т.д. — строго по тем полям, что отдаёт API) |
| `sync_runs` (опционально) | Время запуска, статус, ошибка, сколько строк записано — для мониторинга и отладки                                                                                                                             |

**Требования:**

- **Идемпотентность:** повторный запуск за тот же день **не дублирует** строки — уникальный ключ вида `(snapshot_date, neon_project_id)` или upsert.
- **Временная зона:** все даты в БД — **UTC**; в UI — опциональный выбор отображения (по умолчанию UTC или локаль пользователя).
- **Миграции:** Prisma (или выбранный ORM по TECH_CARD проекта) + версионирование схемы.

---

### 4. Сбор данных (1 раз в день)

- **Триггер:** **Vercel Cron** (например, `0 2 * * *` — 02:00 UTC) или эквивалент для Next.js Route Handler `/api/cron/sync-neon-usage`.
- **Защита cron:** секрет в заголовке (`CRON_SECRET`) или встроенная проверка Vercel Cron, чтобы никто посторонний не дергал эндпоинт.
- **Логика job (один проход в сутки):**
  1. Запрос к Neon API за **вчерашний календарный день** (или за согласованное «скользящее» окно — зафиксировать в реализации).
  2. Нормализация ответа → запись/обновление в `usage_snapshots` (+ обновление кэша `projects`).
  3. Запись строки в `sync_runs` (успех/ошибка).
- **Ретраи:** при временных сбоях API — ограниченное число повторов с backoff; при окончательной ошибке — алерт (см. п. 7).
- **Ручной backfill (опционально, этап 2):** админ-кнопка или скрипт «догрузить период с даты X по Y» с теми же правилами идемпотентности.

---

### 5. Backend / API приложения (Vercel)

- Эндпоинты только для чтения агрегатов для UI, например:
  - список проектов;
  - ряд по дням за диапазон;
  - агрегация по месяцам (можно считать в SQL или в приложении).
- **Не** отдавать `NEON_API_KEY` на клиент; ключ только server-side (Vercel Environment Variables).
- Валидация query-параметров (диапазон дат, лимиты).

---

### 6. Дашборд (UI)

**Функционал:**

- Таблица/карточки проектов с **итогом за период** (выбор: последние 7/30 дней, месяц, кастомный диапазон).
- **Графики:** по дням — линии/столбцы по ключевым метрикам; переключение метрики; при необходимости — stacked по проектам.
- **Фильтры:** проект(ы), период, группировка день/месяц.
- Состояния: загрузка, пустые данные («ещё не было синка»), ошибка API.

**Дизайн (требование «очень красивый, современный»):**

- Тёмная или светлая тема с **чёткой типографикой** и **дизайн-токенами** (CSS variables / Tailwind theme).
- Сетка дашборда (карточки KPI + основной график + таблица детализации).
- Доступность: контраст, фокус, подписи осей, легенда.
- Опционально: экспорт CSV за выбранный период.

Стек UI согласовать с TECH_CARD (например Next.js App Router + Tailwind + библиотека графиков).

---

### 7. Безопасность и эксплуатация

- Секреты только в **env** Vercel; в репозиторий не коммитить.
- Доступ к дашборду: минимум **базовая защита** (Clerk/пароль/один admin user через Vercel Protection — на выбор в реализации).
- Логирование ошибок синка (без утечки ключей); при желании — Sentry.
- Алерт при неуспешном `sync_runs` N дней подряд (email/Slack — этап 2).

---

### 8. Нефункциональные требования

- Язык кода и комментариев: **English**; интерфейс дашборда — по желанию RU/EN.
- TypeScript strict, без `any` на границах — по правилам репозитория.
- Лимиты Vercel: job не должен превышать timeout; при большом числе проектов — пагинация запросов к Neon API и батч-запись в БД.

---

### 9. Критерии приёмки

1. Раз в сутки по расписанию выполняется синк, в БД появляются/обновляются строки за соответствующий день по каждому проекту.
2. Дашборд показывает те же метрики, что приходят из API (согласованность с консолью Neon в пределах задержки и гранулярности API).
3. Повторный cron за тот же день не создаёт дубликатов.
4. Ключи и cron защищены; в клиентском бандле нет секретов.

### 10. Этапы внедрения (рекомендация)

1. **Этап 1 (MVP / foundation) — выполнено:** схема БД (Prisma), cron `/api/cron/sync-neon-usage`, запись «вчера» (UTC), дашборд с графиками и таблицами, read-only API usage, скрипты backfill/reconcile, опциональная защита паролем. См. `docs/01-ARCHITECTURE.md`, `docs/TECH_CARD.md`.
2. **Этап 2 (v1+):** углублённые фильтры и UX, месячные агрегаты (где ещё не закрыто ТЗ), полировка empty/error states, экспорт CSV (по ТЗ п.6), алерты при серии неуспешных sync (п.7), при необходимости — e2e и расширенный мониторинг.
3. **Этап 3 (v2):** расширенный backfill из UI, тёмная тема polish, интеграции алертов (email/Slack).

**Качество и правила Cursor:** линт ESLint, формат Prettier, unit-тесты (Vitest), Husky (`pre-commit` / `pre-push`), CI — см. `docs/QUALITY_AUTOMATION_PLAN.md` и скрипты `pnpm run ci`, `pnpm test:ci`.
